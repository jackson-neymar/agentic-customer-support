# Spec 模板与检查清单

当需要起草、审查或冻结 Spec 时，读取这份参考材料。

## 全新功能提示词

```text
我要开发一个 [功能名称]，核心需求是：[一句话描述]。

请帮我生成一份完整的实现 Spec，包含：
1. 功能概述
2. Given-When-Then 行为描述
3. 边界条件和异常处理
4. API 或接口定义，包括输入、输出和错误码
5. 数据模型或状态变化，如涉及
6. 权限、安全和隐私约束
7. 验收标准和测试计划
8. 实现前必须确认的待确认问题
```

## 已有功能改造提示词

```text
我需要改造现有的 [功能名称]。改动点是：[描述变更]。

请基于当前代码库生成改造 Spec：
1. 说明现有行为和目标行为的差异
2. 识别受影响的模块、API、数据和兼容性风险
3. 给出更新后的 Given-When-Then 场景
4. 说明迁移、向后兼容或灰度发布要求
5. 列出回归测试和手工检查点
6. 标出会影响实现选择的待确认问题
```

## Bug 修复提示词

```text
当前有一个 Bug：[问题描述]。
复现步骤：[步骤]。

请生成修复 Spec：
1. 描述期望行为和实际行为
2. 根据已有证据分析可能根因
3. 明确修复后的验收标准
4. 列出需要新增或更新的测试
5. 说明修复是否可能影响相关功能
```

## 实现前检查清单

- 能否用一句话说清楚目标？
- Given-When-Then 是否覆盖主要成功路径？
- 是否列出了重要失败路径和边界条件？
- 输入格式、允许值、限制和校验规则是否明确？
- 输出、错误码和状态变化是否明确？
- 权限、安全、隐私和数据一致性约束是否明确？
- 是否定义了验收标准和测试点？
- 未决问题是否与假设分开记录？
- 另一位开发者不读代码，是否也能理解目标行为？

## Spec 质量追问

| 维度 | 追问 |
| --- | --- |
| 边界条件 | 到达限制、空状态、过期状态或重复尝试时会怎样？ |
| 并发 | 两个请求同时操作同一资源时会怎样？ |
| 幂等性 | 重复点击、重试或重复请求是否会产生重复副作用？ |
| 数据校验 | 哪些格式、范围和必填字段是有效或无效的？ |
| 安全 | 如何防止滥用、越权访问、注入或敏感信息泄露？ |
| 用户体验 | 加载、空数据、成功和失败状态下，用户应该看到什么？ |
| 验收 | 什么证据能证明任务已经完成？ |

## 常见问题

### Spec 质量不够

问题：AI 会用看似合理的猜测补全缺失的业务规则。

应对方式：

- 先澄清核心流程、边界场景和验收标准。
- 使用 Given-When-Then 降低歧义。
- 区分产品需求和技术约束。
- 要求 AI 暴露不确定点，而不是静默猜测。

### 任务粒度不合适

问题：任务太大则难以验证，任务太小则失去业务语义。

应对方式：

- 按可测试的功能单元拆分。
- 让每个任务聚焦在一个模块或一条业务链路。
- 明确任务之间的依赖关系。
- 可以让 AI 提出拆分方案，但由人确认范围。

### 省略 Review

问题：生成的代码看似完整，却可能遗漏业务语义、失败路径、安全或性能问题。

应对方式：

- 对照 Spec Review，而不是只看代码能否运行。
- 重点检查异常路径、权限、数据一致性和并发。
- 使用测试、类型检查、静态检查和安全检查辅助验证。
- 要求行为变更同时配套测试。

### 协作节奏失控

问题：AI 辅助开发会快速修改大量文件，增加合并冲突风险。

应对方式：

- 以小的、已验证的单元提交或建立检查点。
- 频繁同步，并明确模块边界。
- 对未完成功能使用 feature flag。
- 合并前运行自动化检查。

## 冻结 Spec 提示词

当 Spec 已确认时，可以使用：

```text
以上 Spec 已确认。请基于此开始实现。中途如果发现 Spec 与现有代码冲突，或存在未解决的歧义，请先暂停并说明问题，不要擅自扩大范围。
```
